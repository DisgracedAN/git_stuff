# Git Stuff

Этот репозиторий был создан как упражнение по работе с Git.  
Исходная задача — привести его в «человеческий вид», не переписывая историю коммитов.

## Что было сделано

В рамках приведения репозитория к порядку были выполнены следующие действия:

1.  **Добавлен `.gitignore`**  
    Файл для игнорирования временных и объектных файлов (`*.o`, `*.exe`, `*.log`, папка `build/`), чтобы они не засоряли репозиторий.

2.  **Интегрировано поддерево (`subtree`)**  
    Ветка `add_subtree` была добавлена в основную ветку как поддиректория `subtree/` с помощью команды `git subtree add`.

3.  **Очистка веток**  
    Удалены устаревшие и уже неиспользуемые ветки: `add_subtree`, `first_review`, `master`, `fix_joes_branch`.

4.  **Обновление ветки `joes_branch`**  
    Ветка `joes_branch` была синхронизирована с `main` через `git merge` (без использования `--force`, чтобы не сломать CI).

5.  **Создание тегов**  
    Для ключевых точек истории добавлены теги `v1.0` и `v1.1`.

6.  **Исправление кода**  
    Внесены небольшие улучшения в `main.c` (добавлены запятая и перевод строки для корректного вывода).

## Использованные команды Git

В процессе работы были задействованы следующие команды:

- `git commit` — для создания новых коммитов (`.gitignore`, правки кода).
- `git subtree add` — для интеграции внешнего поддерева.
- `git merge` — для слияния изменений между ветками.
- `git tag` — для маркировки важных версий.
- `git log`, `git blame`, `git stash` — для анализа состояния репозитория.
- `git push` — для отправки изменений в удалённый репозиторий.

Команда `git rebase` **не использовалась**, чтобы сохранить исходные хэши коммитов, как того требовало условие задачи.

## Выполненные шаги и обоснование команд

### 1. Создание `.gitignore` и коммит  

echo "build/" >> .gitignore  
echo "*.o" >> .gitignore  
echo "*.exe" >> .gitignore  
echo "*.log" >> .gitignore  
git add .gitignore  
git commit -m "Added .gitignore to exclude temporary files like .exe .log etc"  
Обоснование:

echo и перенаправление >> – стандартный способ добавить строки в файл из командной строки.

git add – подготавливает файл к коммиту.

git commit -m – создаёт новый коммит с сообщением. Флаг -m позволяет указать сообщение прямо в команде, избегая открытия редактора.

#### Почему так, а не иначе?

Можно было бы использовать git commit -a, но он добавил бы все изменённые файлы, что нежелательно – нужно было добавить только .gitignore.

Не используем --amend, потому что это изменило бы хэш предыдущего коммита, нарушив требование сохранить историю.

### 2. Интеграция поддерева (git subtree add)

git subtree add --prefix=subtree origin/add_subtree -m "Add subtree from add_subtree"  
Обоснование:

git subtree add – команда для включения внешнего репозитория (или ветки) как подкаталога в текущую ветку.  

--prefix=subtree – указывает, в какую папку поместить содержимое (папка subtree/ в корне).  

origin/add_subtree – источник (удалённая ветка).  

-m "..." – задаёт сообщение для создаваемого merge-коммита.  

#### Почему так, а не иначе?  

Альтернатива – использовать git merge --allow-unrelated-histories, но тогда история поддерева смешалась бы с основной, что менее чисто. subtree add создаёт отдельный коммит слияния, который явно показывает, что это внешний код.  
### 3. Удаление устаревших веток  
git push origin --delete add_subtree  
git push origin --delete first_review  
git push origin --delete master  
git push origin --delete fix_joes_branch  
Обоснование:

git push origin --delete <branch> – удаляет ветку на удалённом репозитории.

#### Почему удаляем?   
Ветки add_subtree (уже добавлена как поддерево), first_review (слита с main), master (содержит устаревший коммит с опечаткой) и fix_joes_branch (слита с joes_branch) больше не нужны – они засоряют список и усложняют навигацию.

Удаление веток не удаляет сами коммиты – они всё ещё доступны по хэшам, так что история не теряется.

Альтернатива: можно было бы оставить их, но тогда репозиторий выглядел бы неопрятно. Так как было стремление к «человеческому виду», удаление оправдано.

### 4. Обновление ветки joes_branch (без --force)

git checkout -b joes_branch origin/joes_branch  
git merge main -m "Merge main into joes_branch (synchronization)"  
git push origin joes_branch  
Обоснование:

git checkout -b joes_branch origin/joes_branch – создаёт локальную ветку, отслеживающую удалённую.  

git merge main -m "..." – сливает изменения из main в текущую ветку (joes_branch), создавая merge-коммит с заданным сообщением.  

git push origin joes_branch – отправляет изменения на удалённый репозиторий без --force.  

#### Почему так, а не иначе?

Использовать rebase для обновления joes_branch было бы проще, но это переписало бы историю, что запрещено.

Использовать merge – безопасно, т.к. добавляется только новый коммит, а существующие коммиты не затрагиваются.

Не использовать --force – чтобы не сломать CI, о чём прямо сказано в README оригинального репозитория.

### 5. Создание тегов

git tag -a v1.0 -m "First working ver" 3cec852  
git tag -a v1.1 -m "After the review" f9568cc  
Обоснование:

git tag -a – создаёт аннотированный тег (содержит метаданные: автор, дата, сообщение).

-m "..." – задаёт сообщение для тега.

Указание хэша коммита (3cec852 и f9568cc) – привязывает тег к конкретному коммиту.

Почему аннотированный, а не лёгкий? Аннотированный тег хранит дополнительную информацию и считается лучшей практикой для релизных версий.

Зачем теги? Они позволяют легко находить важные вехи в истории, не запоминая хэши.

### 6. Исправление кода в main.c

notepad main.c  
git add main.c  
git commit -m "Изменён main.c- добавлен перенос строки и запятая"  
Обоснование:  

Редактирование файла вручную – стандартный способ внесения изменений.

git add – добавляет изменённый файл в индекс.  

git commit -m – создаёт новый коммит с осмысленным сообщением.  

Почему не git commit -a? Чтобы явно указать, какой именно файл меняется, и избежать случайного включения других правок.  

### 7. Проверка состояния (git stash, git log, git blame)  
git stash list          # убедились, что нет скрытых изменений  
git log --oneline --graph --all --decorate   # визуализация истории  
git blame main.c        # проверка авторства строк (нет ошибок, требующих revert)  
Обоснование:  

git stash list – показывает, есть ли временно сохранённые изменения. Они отсутствовали, поэтому команда не потребовалась.  

git log с флагами --oneline (краткий формат), --graph (визуализация ветвления), --all (все ветки), --decorate (показывает ссылки) – даёт полную картину истории, помогает убедиться, что структура стала чище.  

git blame main.c – показывает, кто и когда изменял каждую строку. Мы не нашли проблемных мест, поэтому git revert не применялся.  

### 8. Пуш изменений

git push origin main  
git push origin joes_branch  
git push origin --tags  
Обоснование:  

Обычный git push (без --force) – безопасная отправка новых коммитов и тегов.  

--tags – отправляет все локальные теги на удалённый репозиторий.  

Почему без --force? Мы не переписывали историю, поэтому --force не нужен. Для joes_branch он даже запрещён условием.  

### Какие команды не использовались и почему

git rebase – Запрещён условием – переписывает историю, меняя хэши коммитов.  
git revert – Не потребовался – все коммиты корректны, нет необходимости отменять изменения.  
git stash apply – stash list был пуст, поэтому применять было нечего.  
#### Альтернативные подходы (если бы разрешалось менять историю)  
В иных обстоятельствах можно было бы:

Использовать git rebase -i для объединения коммитов Better, Best, Even better в один осмысленный, а также переименовать неинформативные сообщения.  

Применить git commit --amend для правки последнего коммита.  

Задействовать git filter-branch для полного удаления бинарных файлов из всей истории.  

Однако по условию задачи это не допускалось, поэтому выбраны только неразрушающие операции.
